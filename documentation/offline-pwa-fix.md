# Offline PWA Fix — 2026-05-07

## Symptom

The fork was served on the LAN via Caddy + PHP-FPM on port 8445 and installed
as a PWA on a phone. It worked offline for roughly one day, then stopped —
the page hung trying to reach the server.

## Root cause

The service worker in this fork (`dist/sw.js`) had been reduced to:

```js
var urlsToCache = [".", "css/style.css"];
```

Only the document root and one CSS file were pre-cached. The fetch handler
used a "cache, then network" strategy, so every other asset (jquery,
`cstimer.js`, `twisty.js`, the icon, the webmanifest, language files) was
served from the network on each request.

For about a day this *appeared* to work because the **browser's HTTP cache**
held those assets via heuristic freshness (Caddy's `file_server` sets no
`Cache-Control` headers, so the browser falls back to roughly 10% of the
file's `Last-Modified` age). Once heuristic freshness expired, the browser
revalidated → Caddy was no longer running → the SW fell through to a failed
network request → the app broke.

The legacy `<html manifest="cache.manifest">` AppCache mechanism did not
help either: AppCache was removed from Chrome (95+) and Firefox (85+), and
the `cache.manifest` in this fork was empty (only an MD5 stub line).

The reduction to two URLs came from commit `11e1eea` ("adapt for android
compatibility"), which replaced `timer.php` as the served entry point with a
static `index.html` and trimmed both the SW pre-cache list and the AppCache
manifest.

## Fix

### `dist/sw.js`

Restored the full pre-cache list and made the fetch handler properly
offline-capable:

- Pre-cache all core assets at install: `./`, `index.html`, `css/style.css`,
  `js/jquery.min.js`, `js/cstimer.js`, `js/twisty.js`, `cstimer512x512.png`,
  `cstimer.webmanifest`.
- `skipWaiting()` on install and `clients.claim()` on activate, so a new SW
  takes over immediately rather than waiting for all tabs to close.
- Cache-first fetch handler: serve from cache; on miss, fetch and store any
  successful same-origin GET (so language files like `lang/de-de.js` get
  cached the first time they are loaded).
- On network failure for navigation requests, fall back to the cached `./`.

### `dist/timer.php`

Removed `manifest="cache.manifest"` from the `<html>` tag. AppCache is dead
in modern browsers and was actively confusing the situation. Since
`dist/index.html` is regenerated from `dist/timer.php` by `make`, the fix
must live in the PHP source.

### `Makefile`

Removed the `$(dest)/cache.manifest` rule and dropped it from the `all:`
target. The companion file `dist/cache.manifest` was deleted.

The `$(dest)/sw.js` rule remains: it deletes the last line of `dist/sw.js`
and appends `var CACHE_NAME = "cstimer_cache_<md5>";`, where the MD5 is
computed from the cached asset list. This keeps cache invalidation
automatic — any change to a precached asset bumps `CACHE_NAME`, which
triggers re-install on the client. Because the rule rewrites only the last
line, `sw.js` was structured to keep the `var CACHE_NAME = ...` declaration
on the final line.

### `src/` cleanup

`src/sw.js`, `src/index.php`, and `src/cache.manifest` were vestigial
upstream files unused by the Makefile (which treats `dist/sw.js` and
`dist/timer.php` as the source-of-truth). They were deleted to remove
ambiguity about which file is the real source.

## Notes on `dist/` as source

The `dist/` directory in this fork is a mix of build outputs and
source-of-truth files:

- **Generated by `make`:** `index.html` (from `timer.php`), `js/cstimer.js`,
  `js/twisty.js`, `css/*`, `lang/*`.
- **Source-of-truth, edited in place:** `sw.js` (last line patched by the
  Makefile, body otherwise preserved), `timer.php`, `cstimer.webmanifest`,
  `cstimer512x512.png`.

This means edits to `dist/sw.js` (above the last line) and `dist/timer.php`
persist across builds and should be made directly there, not in `src/`.

## Deploying the fix to an installed PWA

Existing installations are running the old service worker. Procedure:

1. Start Caddy so the device can reach the server once.
2. Open the PWA. The browser fetches the new `sw.js`, sees a different
   `CACHE_NAME`, runs install (pre-caching all assets), then activates and
   deletes the old cache.
3. From this point the app runs entirely from the SW cache and no longer
   depends on Caddy being reachable.
